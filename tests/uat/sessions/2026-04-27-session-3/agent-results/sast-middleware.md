### SEV-2 — Duplicate `http.response.start` if downstream begins responding before body cap fires
- **Description.** `BodySizeCapMiddleware._send_413` sends a fresh `http.response.start` after `_BodyTooLargeError` propagates from the downstream app. ASGI requires exactly one `http.response.start` per request. BL5 has no body-reading endpoint today, so this is latent — but the design contract documented in ADR-0012/ID7 ("intercept receive() and send 413") fails the moment a future endpoint streams a response while still draining the request body. A handler that does `async for chunk in request.stream(): ...` and writes a streaming response (e.g., a future SSE / NDJSON chunk endpoint) will have already emitted `http.response.start` before the cap fires; the middleware then emits a second start frame, and the underlying server (uvicorn) will either log a `RuntimeError` and tear the connection down or, worse, write the second start as headers in the body of the first response, leaking 413 framing into the response body.
- **Concrete exploit / scenario.** Once BL6+ ships any endpoint that interleaves `await request.stream()` with `StreamingResponse`, an attacker sends a chunked POST whose first chunks yield a partial response and whose later chunks push total bytes over 32 KiB. Result on a permissive server: response framing corruption visible to legitimate clients sharing the same keep-alive connection (HTTP smuggling-adjacent surface). On a strict server: 500 with traceback in logs, plus a forced disconnect that resets the connection on a per-request basis (not a request-amplifier, but DoS-adjacent because it dirties the keepalive pool).
- **Affected code.** `src/orchestrator/api/middleware.py:148-157` (the `try/except _BodyTooLargeError` block) and `:159-173` (`_send_413`).
- **Suggested fix.** Track whether a `http.response.start` has already been sent by wrapping `send` in `BodySizeCapMiddleware` with a `started: bool` flag; if `started` is True when `_BodyTooLargeError` fires, do **not** call `_send_413` — instead log `api.body_size_cap_exceeded_after_response_started` at ERROR and either propagate the exception (forcing the server to close the connection) or send only `{"type": "http.response.body", "body": b"", "more_body": False}` to terminate. Document in ADR-0012 that the streaming cap is best-effort once response framing has begun.
- **Regression test sketch.** Direct middleware unit test where `downstream_app` first calls `send({"type": "http.response.start", "status": 200, ...})`, then drains body until `_BodyTooLargeError` raises; assert exactly one `http.response.start` appears in `sent_messages`.

### SEV-2 — `BearerAuthMiddleware` 401 / 413 / 403 responses lack CORS headers (browser failure mode)
- **Description.** The middleware ordering puts `CORSMiddleware` *innermost*. When `BearerAuthMiddleware._send_401` (or `_send_403`, or `BodySizeCapMiddleware._send_413`) shortcuts the response, the `CORSMiddleware` is never invoked, so `Access-Control-Allow-Origin` and friends are absent. A browser making a same-origin request is unaffected; a browser issuing an authenticated cross-origin request (Game_shelf is configured via `cors_origins`) will see the 401 as a "CORS error", not as an authentication failure, masking the real problem and breaking error UX. With `cors_origins=[]` (default) this is moot, but the moment Game_shelf is added it bites.
- **Concrete exploit / scenario.** Operator deploys Game_shelf at `http://gameshelf.lan` and adds it to `cors_origins`. A user's session token expires; the next API call returns 401 without ACAO; the browser surfaces "blocked by CORS policy" instead of "401 Unauthorized"; Game_shelf cannot detect the auth failure to redirect to re-auth. Not a vulnerability per se, but it actively defeats the response-correctness contract that BL6 will rely on.
- **Affected code.** `src/orchestrator/api/main.py:102-112` (middleware registration order); `middleware.py:256-289` (`_send_401`, `_send_403`).
- **Suggested fix.** Move `CORSMiddleware` to outermost (register it *last* so it wraps everything). Re-verify preflight still works: an `OPTIONS` request would then hit CORS first (handles preflight, returns 204 with ACAO) — which matches Starlette's documented usage. The current ordering rationale was "CORS innermost so preflight bypasses auth", but `BearerAuthMiddleware` already has `if method == "OPTIONS": pass-through` at line 194-196, so CORS doesn't need to run before auth to handle preflight; it needs to run *outside* auth so error responses get ACAO.
- **Regression test sketch.** `await client.get("/api/v1/games", headers={"Origin": "http://gameshelf.lan"})` (no auth) — assert `r.status_code == 401` AND `r.headers.get("access-control-allow-origin") == "http://gameshelf.lan"`.

### SEV-3 — `BearerAuthMiddleware` reveals `path` in the rejection log on bad-token / non-loopback paths (TM-013-adjacent)
- **Description.** TM-013's mitigation strategy says auth runs at the middleware layer specifically so that 401 vs 404 cannot be distinguished by an unauthenticated client. The HTTP response itself is uniform (always 401 with `{"detail":"unauthorized"}` regardless of path), which is correct. **However**, the rejection log line `api.auth.rejected` includes the `path=path` field. An attacker who has obtained log-stream access (e.g., the Docker stdout sidecar, Game_shelf compromise, or a stolen JIT log share) can enumerate which paths exist and are protected vs. exempt, partially undoing TM-013. In particular `LOOPBACK_ONLY_PATTERNS` triggers a *different* log event (`reason="non_loopback"`) than a missing-header rejection on the same path, distinguishing "this path requires loopback" from "this path requires a token" by log-channel observation alone. The rejection_fingerprint sha256[:8] also lets a log-reading attacker correlate retry attempts with the same wrong token.
- **Concrete exploit / scenario.** TM-023 says Game_shelf is the pivot. An attacker who has read-only log access (not full RCE) on the Docker host can issue requests from a non-loopback IP to `/api/v1/platforms/steam/auth` with an invalid token vs. valid token vs. no token, observe log lines, and infer (a) the path exists and is loopback-restricted and (b) whether their candidate token matched. The path leak is low value (paths are documented in OpenAPI which is exempt anyway), but the differential `reason=` fields and `rejection_fingerprint` are unnecessary surface.
- **Affected code.** `src/orchestrator/api/middleware.py:207, 212, 218, 231-236, 245-250`.
- **Suggested fix.** Either (a) confirm in the threat model that log access ≈ token access ≈ host compromise (in which case this is acceptable per the doc's own residual-risk language), or (b) collapse the four `api.auth.rejected` reasons into a single un-typed event with no `path=` and no `rejection_fingerprint=` for the no-header / malformed-header / bad-token cases. Keep the fingerprint only on `reason="bad_token"` to support legitimate rotation debugging, and gate it behind a `LOG_AUTH_FINGERPRINT=1` env var off by default.
- **Regression test sketch.** Capture stdout JSON during three rejection paths (no header / bad header / bad token / non-loopback on protected path) and assert all four produce log entries that are byte-identical except for `correlation_id` and `timestamp`.

### SEV-3 — `BodySizeCapMiddleware` does not enforce against a lying client (CL ≠ actual body size)
- **Description.** When a client sends `Content-Length: 100` but actually streams 64 KiB before the server signals end-of-body, the proactive CL path passes (100 ≤ cap), and the streaming path is only consulted if the downstream app calls `receive()`. For BL5 there is no body-reading endpoint, so this is latent. But the design statement is "32 KiB cap"; the implementation is "32 KiB cap *if either CL is honest or the handler reads the body*". The streaming cap relies on the handler draining; an early-returning handler will never trigger receive() and the body silently sits in the server's read buffer.
- **Concrete exploit / scenario.** BL6+ ships `POST /api/v1/games/steam/{id}/prefill` with no request body but `Content-Length: 0`. An attacker sends `Content-Length: 0` plus 100 MiB of body bytes. The handler returns 202 immediately without touching the body. Whether those 100 MiB sit in uvicorn's buffer or are flushed depends on the server; on a slow link this can be used as a slowloris-style memory hog by stacking many such requests against the connection pool. Mitigation already exists at the LAN-only trust boundary (TM-015), so this is informational rather than urgent.
- **Affected code.** `src/orchestrator/api/middleware.py:111-149`.
- **Suggested fix.** Document in ADR-0012 that the 32 KiB cap is best-effort against honest clients, and is hardened by uvicorn's `h11_max_incomplete_event_size` plus the LAN-only network boundary. If full enforcement is required, force a `await request.body()` drain in a low-level dependency or wrap the receive in a way that drains greedily even when the handler ignores the body.
- **Regression test sketch.** Send `Content-Length: 0` with a 64 KiB body to a no-body-reading endpoint; assert connection is reset *or* request is rejected, not silently accepted.

### SEV-3 — `_lifespan` does not call `close_pool` if `init_pool` fails partway
- **Description.** In `main.py:_lifespan`, if `init_pool()` raises after it has already opened the writer connection but before all readers are open, `init_pool` itself is responsible for cleanup of partially-initialized state. The lifespan handler does NOT call `close_pool` on the SystemExit(1) path — the `yield` is never reached, so the post-yield cleanup never runs. If `init_pool` does not have a `try/except → cleanup → re-raise` pattern, partially-initialized pool state (open connections, locked WAL files) leaks until process exit. Process exit happens (SystemExit(1)), so file descriptors are reclaimed by the OS, but any aiosqlite background task or thread launched by partial init may continue briefly past the SystemExit raise on the asyncio loop until the loop tears down.
- **Concrete exploit / scenario.** Migration succeeds, init_pool opens writer, the first reader fails to open (e.g., disk full mid-init). `init_pool` re-raises `PoolError`; lifespan logs `pool_init_failed` and raises SystemExit(1); writer connection is left to be reaped by GC during interpreter shutdown. If a background BL6+ task was spawned during init (none today, but possible in future), it sees the loop being torn down and may emit a confusing CRITICAL log line about loop closure. Not an exploitable defect, but a fragility.
- **Affected code.** `src/orchestrator/api/main.py:53-58, 66-72`.
- **Suggested fix.** Restructure as `try / except / finally` so that on init_pool failure, an explicit `await close_pool()` runs to drain whatever did successfully open. Alternative: rely on `init_pool` to be transactional (open-all-or-nothing), and document that contract. Given BL4 just landed, verify which contract it provides and align.
- **Regression test sketch.** Patch the BL4 reader-open path to raise on the 2nd reader; lifespan startup; assert `init_pool` called once, `close_pool` called once, lifespan exits with SystemExit(1).

### SEV-3 — Health endpoint Pydantic model exposes `git_sha` unconditionally (TM-013 deferred)
- **Description.** The threat model's TM-013 mitigation explicitly says: "Phase 3 hardening: make `git_sha` conditional on bearer auth; leave `version` + `status` unauthenticated for Game_shelf health checks." BL5 ships `git_sha` unconditionally on the public `/api/v1/health` response (`routers/health.py:30, 70`). Per the threat model, this is acceptable for Phase 2 ("low priority for LAN-only single-user") and tracked for Phase 3, but it should be *recorded* as a known TM-013 residual until the conditional-disclosure landing.
- **Concrete exploit / scenario.** Per TM-013: an attacker on the LAN reads `git_sha`, looks up the orchestrator's exact build commit on GitHub, identifies dependency versions present at that commit, searches NVD for CVEs in those versions. LAN-only trust boundary makes this low-risk in practice.
- **Affected code.** `src/orchestrator/api/routers/health.py:30, 70`.
- **Suggested fix.** Either (a) confirm explicit deferral to Phase 3 in `BUGS.md` (probably already tracked) and accept; (b) change the `health` route to omit `git_sha` when no Bearer token is present in the request — but this requires reading auth state at the router, which contradicts the "auth-at-middleware" design unless the middleware sets `scope["state"]["authenticated"] = True/False` non-blockingly. Option (a) is the cheaper path.
- **Regression test sketch.** None for BL5 (deferred); for Phase 3, assert `/api/v1/health` without auth returns body without `git_sha` key.

### SEV-4 — `BodySizeCapMiddleware` accepts negative or non-integer Content-Length silently as 0
- **Description.** `int(cl_bytes)` raises `ValueError` on `Content-Length: abc`; the `except ValueError: cl = 0` swallows it. A negative CL (`Content-Length: -1`) parses successfully to `-1`, which is `< self.cap`, so the proactive path lets it through. The streaming path then runs and works correctly (it counts actual bytes). Negative CL is invalid HTTP per RFC 9110 §8.6 — the server should reject the request, not pass it through. uvicorn may already reject this at the parser layer; verify.
- **Concrete exploit / scenario.** Client sends `Content-Length: -1` with 64 KiB body. If uvicorn's parser accepts this (some embedded servers do), the proactive cap is bypassed and protection falls back to streaming, which works only if the handler drains. Combined with the SEV-3 lying-client issue above, this is a hardening gap rather than a vuln.
- **Affected code.** `src/orchestrator/api/middleware.py:120-124`.
- **Suggested fix.** Reject `cl < 0` at the proactive path: `if cl < 0: await self._send_400(send); return`. Treat malformed `Content-Length` (the `ValueError` branch) as a 400 too, not a silent pass-through.
- **Regression test sketch.** `await client.post("/...", headers={"Content-Length": "-1"}, content=b"x"*100)` → assert 400.

### SEV-4 — `CorrelationIdMiddleware` UUIDv4 regex permits a forced-collision attack on log indexing
- **Description.** The middleware accepts any client-supplied X-Correlation-ID that matches UUID4-shape and reuses it as the canonical ID. A client can send the same UUID4 on every request, causing all of that client's requests to share a correlation_id, which can confuse log aggregation and hide attack patterns from a `correlation_id`-keyed search. Not a vulnerability, but it weakens the diagnostic value of correlation IDs as an audit primitive (TM-008/TM-009 mention `source` columns and logs as repudiation defense).
- **Concrete exploit / scenario.** Attacker scripts 1000 requests with `X-Correlation-ID: 11111111-2222-4333-8444-555555555555`; all 1000 share the ID; an investigator searching for that ID sees a clump of 1000 events all marked as one logical "request flow", muddying timeline reconstruction.
- **Affected code.** `src/orchestrator/api/middleware.py:64-67`.
- **Suggested fix.** Either (a) accept the client value but ALSO emit a server-generated `request_id` field per request, so logs always carry a unique-per-request key; or (b) reject client-supplied IDs entirely and always generate. Spec §5.2 likely requires (a) for cross-system tracing — verify which.
- **Regression test sketch.** Two sequential requests with identical client-supplied `X-Correlation-ID`; assert log lines have a `request_id` field that differs between the two.

## Investigation summary by topic

**A. Middleware ordering effects.** The stack outermost→innermost is CorrelationId → BodySizeCap → BearerAuth → CORS. CorrelationId observes raw scope and wraps `send` to append the CID header to whatever response (401, 413, 200, 5xx) ultimately escapes. BodySizeCap observes raw scope and wraps `receive` for streaming-cap interception. BearerAuth observes raw scope and short-circuits 401/403 without invoking inner CORS. CORS handles browser preflight on OPTIONS. **Wrong outcomes:** (1) 401/413/403 responses lack ACAO headers (SEV-2 above) — CORS should be outermost. (2) BodySizeCap's `_send_413` can produce a duplicate `http.response.start` if a future streaming handler has already begun responding (SEV-2 above).

**B. TM-013 fingerprinting defense.** Auth runs at ASGI middleware, so unauthenticated clients cannot observe 404 vs. 401 in HTTP responses. Verified the three rejection paths produce byte-identical 401 bodies/headers. **However**, there is a non-HTTP fingerprinting channel via the structured log: `api.auth.rejected` carries a per-rejection-reason `reason=` field plus a `path=` and `rejection_fingerprint=` (SEV-3 above). Acceptable if log access is treated as host compromise; surface in BUGS.md if not.

**C. TM-018 streaming body cap.** (1) ✅ CL > cap rejected proactively; CL = cap+1 rejected. (2) ✅ Streaming path correctly accumulates `bytes_received` and raises `_BodyTooLargeError` when the running sum exceeds cap. (3) ✅ Chunked / no-CL requests reach the streaming path. (4) ✅ CL=0 / CL=cap pass; CL=cap+1 rejected. CL absent → streaming path. (5) ⚠️ Lying client sends CL < actual body — proactive path is bypassed; streaming path enforces only if the handler drains the body (SEV-3). (6) ⚠️ Negative CL silently treated as 0 (SEV-4).

**D. Lifespan failure paths.** Migrations raise → SystemExit(1), `close_pool` not called (correct, pool not yet initialized). init_pool raises → SystemExit(1), `close_pool` not called even if pool was *partially* initialized (SEV-3 above). `close_pool` raises during shutdown → caught and logged; lifespan exits cleanly. The lifespan's exception flow is sound for the documented happy paths but does not handle partial init cleanup.

**E. ASGI scope handling.** All three middlewares short-circuit `scope["type"] != "http"`, so `lifespan` and `websocket` scopes flow unmodified to the downstream FastAPI app. No middleware indexes `scope["method"]`, `scope["path"]`, `scope["headers"]`, or `scope["client"]` *before* the http-only guard. `scope.get("client")` returning `None` is correctly handled at `middleware.py:242-244`. There are no FastAPI websocket endpoints in BL5 to test, but a manual `scope = {"type": "websocket", ...}` walkthrough confirms no AttributeError is reachable.

**F. New threat candidates** — see next section.

## New threat candidates (beyond TM-001..TM-023)

- **TM-NEW-A — ASGI response framing corruption via duplicate `http.response.start`.** Covered in SEV-2 above. Concrete in BL6+ once a body-reading endpoint ships. Exploit: a streaming POST handler whose `read body → write response chunk → read body` interleaving causes the body cap to fire mid-response. Mitigation: see SEV-2 fix.

- **TM-NEW-B — Cross-origin error masking via missing CORS headers on auth/cap rejections.** Covered in SEV-2 above. Browser-side error UX is broken once Game_shelf is added to `cors_origins`. Exploit: not security per se, but it blocks Game_shelf's ability to detect 401 and trigger re-auth, allowing stale-session conditions to persist longer than they should.

- **TM-NEW-C — Log-channel auth fingerprinting (TM-013 extension).** Covered in SEV-3 above. An attacker with read-only log access (one rung below TM-023's full Game_shelf compromise) can distinguish "path doesn't exist" from "path exists but token wrong" from "path exists but caller not loopback" via the `reason=` field, partially undoing the TM-013 design. Exploit: scripted probe + log scraper.

- **TM-NEW-D — Slow-body / lying-Content-Length resource hold.** Covered in SEV-3 above. A future body-reading endpoint that returns 202 without draining (a fire-and-forget pattern common for prefill enqueue) is open to a client that sends `Content-Length: 0` with a multi-MiB body, holding server read buffers per connection. Combined with uvicorn's default per-connection limits and the LAN trust boundary, this is bounded but not eliminated. Exploit: 256 concurrent slow-body connections × N MiB pending = N × 256 MiB resident on the server.

- **TM-NEW-E — Loopback-spoofing via misconfigured reverse proxy injecting `client=("127.0.0.1", X)`.** The OQ2 enforcement in `BearerAuthMiddleware` checks `scope["client"][0] == "127.0.0.1"`. If the orchestrator is ever placed behind a reverse proxy (Caddy, nginx, Traefik) without `forwarded_allow_ips` configured correctly, *every* request will appear to originate from 127.0.0.1 because `scope["client"]` is the proxy, not the original client. A LAN attacker hitting the proxy could reach the loopback-only `POST /api/v1/platforms/{name}/auth` endpoint despite its protection.
  - **Exploit scenario.** Operator deploys orchestrator behind Caddy on the same Docker network. Caddy's `client_addr` is `127.0.0.1` from the orchestrator's perspective. Attacker on the LAN sends `POST /api/v1/platforms/steam/auth` with a stolen bearer token; bearer check passes, loopback check passes (because `scope["client"][0]` is Caddy, on loopback), endpoint runs from a non-trusted client.
  - **Mitigation.** (1) Document in HANDOFF.md that the orchestrator must be exposed *directly* on its LAN port and never placed behind a reverse proxy without TrustedHostMiddleware + ProxyHeadersMiddleware configured. (2) Optionally check `X-Forwarded-For` is *absent* for loopback-required paths; reject if present (proxy headers indicate the real client is non-loopback).
  - **Severity.** SEV-3, but architectural — flag for ADR + HANDOFF rather than code change.

- **TM-NEW-F — `git_sha` from environment is operator-controlled and propagated unredacted.** `main.py:62` reads `os.environ["GIT_SHA"]` and stores it on `app.state` for echo by the public health endpoint. There is no validation. An operator (or any process that can set the orchestrator's environment) can plant arbitrary content in `git_sha` — including credentials, log-injection payloads (CR/LF), or large strings. The health response then echoes that string verbatim. Exploit: a malicious entry-point script or compose env file can inject `GIT_SHA="$(cat /run/secrets/orchestrator_token)"` and the health endpoint will publicly expose the token. This depends on the operator being malicious or compromised, which is outside the threat model boundary, but it's still a hardening gap that's cheap to close. **Suggested fix.** Validate `GIT_SHA` against `^[0-9a-f]{7,40}$` at lifespan startup; on mismatch, log a WARNING and use `"unknown"`.
